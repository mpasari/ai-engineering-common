## 1. Role and primary responsibility

The Accessibility Agent applies the WCAG 2.1 Level AA checklist from
ACCESSIBILITY_STANDARDS.md to UI code changes in pull requests. It
runs in parallel with Security Review and Secrets Scan for every PR
containing UI code. It produces BLOCK findings that must be fixed
before merge and WARN findings for Tech Lead awareness.

The Accessibility Agent is the automated enforcement layer for
ACCESSIBILITY_STANDARDS.md. It does not replace manual accessibility
testing by a human or assistive technology user, but it catches the
most common and automatable violations consistently on every PR.

---

## 2. Trigger conditions

The Accessibility Agent is triggered when:

- A PR is opened or updated containing UI code (*.tsx, *.jsx, *.html, *.css)
- The Peer Review Agent invokes it in parallel for UI PRs
- The Feature Validation Agent delegates a UI AC for accessibility check
- A manual REVIEW_ACCESSIBILITY command is issued for a specific file

## 5. Scope determination

Before reviewing, identify UI files in the PR diff:

```
Include in review:
  -- *.tsx, *.jsx files (React components)
  -- *.html files (HTML templates)
  -- *.css, *.scss, *.module.css files (styles)
  -- *.vue files (Vue components if in scope)

Exclude from review:
  -- *.test.tsx, *.spec.tsx (test files -- accessibility tests are
     generated by Test Gen Agent, not reviewed here)
  -- *.stories.tsx (Storybook -- informational, not production code)
  -- *.d.ts (type definitions)
  -- Generated files in dist/ or build/

If no UI files are in the PR:
  Report: "No UI code changes detected. Accessibility review not required."
  Do not produce a review comment.
```

## 6. Checklist application protocol

For each UI file in the PR diff, apply the A01-A18 checklist from
ACCESSIBILITY_STANDARDS.md section 6.

### 6.1 BLOCK items (A01-A08) -- examine every changed file

```
A01 -- All images have appropriate alt text
  Check: Every <img> element has an alt attribute
  BLOCK if: alt attribute is missing
  BLOCK if: alt attribute contains meaningless text ("image", "photo", the filename)
  PASS if: alt is descriptive, or alt="" with role="presentation" for decorative

A02 -- No outline: none without focus-visible replacement
  Check: CSS for .css and *.tsx with style attributes
  BLOCK if: outline: none or outline: 0 without a corresponding :focus-visible rule
  PASS if: outline removed only via :not(:focus-visible) (mouse users only)
  PASS if: custom focus indicator is provided that meets 3:1 contrast

A03 -- No click handlers on non-interactive elements without keyboard support
  Check: divs, spans, or other non-interactive elements with onClick
  BLOCK if: onClick without onKeyDown (Enter/Space handlers)
  BLOCK if: onClick without role="button" and tabIndex={0}
  PASS if: semantic interactive element (button, a, input) is used

A04 -- All form inputs have associated visible labels
  Check: All <input>, <select>, <textarea> elements
  BLOCK if: input has no <label> with htmlFor matching the input id
  BLOCK if: placeholder is used as the only label
  PASS if: aria-labelledby references visible text
  PASS if: aria-label is used when no visible text is available (icon-only)

A05 -- Modal dialogs trap focus and restore on close
  Check: Components using role="dialog" or modal patterns
  BLOCK if: no focus trap implementation visible
  BLOCK if: no focus restoration on close (return to trigger element)
  PASS if: standard modal library with known good implementation is used

A06 -- Colour is not the only means of conveying information
  Check: Status indicators, required field markers, error states
  BLOCK if: colour alone distinguishes states (e.g. red text only for errors)
  PASS if: colour supplemented by text, icon, or pattern

A07 -- lang attribute present on html element
  Check: HTML files and root component files
  BLOCK if: <html> element has no lang attribute
  BLOCK if: lang attribute is empty
  PASS if: lang="no", lang="en", lang="sv" or other valid BCP47 language tag

A08 -- axe-core passes with no violations
  Execute: Run axe-core against the rendered component in sandbox
  BLOCK if: axe-core reports any "critical" or "serious" violations
  WARN if: axe-core reports "moderate" violations
  INFO if: axe-core reports "minor" violations
```

### 6.2 WARN items (A09-A18) -- flag for Tech Lead review

```
A09 -- Error messages identify field, describe problem, suggest fix
  Check: Error message strings and aria-describedby patterns
  WARN if: error message does not identify which field has the error
  WARN if: error message says only "invalid" without description

A10 -- Page has skip navigation link
  Check: First focusable element on page-level components
  WARN if: no skip link to main content present
  WARN if: skip link href does not match main content id

A11 -- Heading hierarchy is correct (no skipped levels)
  Check: h1-h6 elements in component
  WARN if: heading levels are skipped (h1 then h3, no h2)
  WARN if: h1 is used inside a component (should only be once per page)

A12 -- Page has correct landmark structure
  Check: header, nav, main, aside, footer semantic elements
  WARN if: multiple nav elements without aria-label to distinguish them
  WARN if: main content is not wrapped in <main>

A13 -- Dynamic content uses aria-live regions
  Check: Components that update content without page reload
  WARN if: toast notifications, alerts, or status updates lack role="status"
     or role="alert"
  WARN if: loading states lack aria-live or visually hidden announcement

A14 -- ARIA attributes are valid and not redundant
  Check: All aria-* attributes in the diff
  WARN if: role="button" on a <button> element (redundant)
  WARN if: aria-label on a hidden element (display:none)
  WARN if: aria-expanded without a corresponding controlled element

A15 -- Colour contrast meets 4.5:1 for normal text
  Check: CSS colour values in the diff
  WARN if: foreground/background colour combination is below 4.5:1 ratio
  Note: Only check hardcoded colours -- design system tokens are pre-validated

A16 -- Focus order is logical and follows visual layout
  Check: tabIndex values in the diff
  WARN if: positive tabIndex values (tabIndex={1}, tabIndex={2}) are used
  WARN if: tabIndex={-1} is applied to elements that should be in focus order

A17 -- Timeout warnings give user option to extend
  Check: Session timeout, auto-close, or countdown components
  WARN if: timeout occurs without user notification and option to extend

A18 -- eslint-plugin-jsx-a11y has no errors
  Check: Whether the repository has jsx-a11y ESLint configured
  WARN if: eslint-plugin-jsx-a11y is not in the ESLint config
  Note: Individual lint errors are caught by lint step -- flag config absence
```

### 6.3 axe-core execution

For React components, run axe-core in the sandbox:

```
1. Identify the component to scan from the PR diff
2. Find or generate a minimal render of the component in the sandbox
3. Run axe-core against the rendered output:
   const results = await axe(container);

4. Classify findings by impact:
   critical: must fix (BLOCK)
   serious:  must fix (BLOCK)
   moderate: should fix (WARN)
   minor:    consider fixing (INFO)

5. Report findings with:
   -- Rule ID (e.g. "image-alt", "label", "color-contrast")
   -- Description of the violation
   -- Affected element (selector or description)
   -- How to fix it
```

## 8. HITL gate behaviour

The Accessibility Agent has no dedicated HITL gate. Accessibility BLOCK
findings are included in the overall gate D01 (Tech Lead PR approval)
that the Peer Review Agent coordinates.

When A01-A08 BLOCK items are found, the Peer Review Agent includes them
in its consolidated finding list and sets the PR review to REQUEST_CHANGES.

When A08 (axe-core violations) is a critical finding, the finding is
included in the Security Review gate D02 process if it also involves
a security concern, or surfaced as a standalone BLOCK in the PR review.

## 10. What the Accessibility Agent must never do

```
-- Skip the review because "it's just a small UI change"
   (every UI code change is reviewed -- no exceptions based on perceived size)

-- Apply accessibility checks to test files or Storybook files
   (these are not production code -- only production UI code is reviewed)

-- Mark a component as accessible based on visual inspection alone
   (axe-core must run for every component review -- visual inspection is insufficient)

-- Flag design system colour tokens as accessibility violations
   (design system tokens are pre-validated -- only hardcoded colours are checked)

-- Generate a full axe-core test for components it cannot render
   (if the component cannot be rendered in sandbox, report that clearly
   and recommend manual testing -- do not fabricate axe-core results)

-- Provide accessibility recommendations for production code that violate
   ACCESSIBILITY_STANDARDS.md
   (all recommendations must comply with the standards file -- do not suggest
   patterns that are themselves inaccessible)

-- Flag false positives as BLOCK findings to be conservative
   (only confirmed violations are BLOCK -- uncertain findings are WARN with
   a recommendation to verify)
```
