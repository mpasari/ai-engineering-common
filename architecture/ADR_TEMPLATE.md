## Guidance for writing good ADRs

### What makes a good ADR

```
Good ADRs:
  -- Document decisions that are hard to reverse (technology choices,
     architectural patterns, integration approaches)
  -- Explain WHY, not just WHAT (the code shows what was decided;
     the ADR explains why)
  -- Are written close to the time of the decision (not reconstructed months later)
  -- Include the options that were rejected and why

Poor ADRs:
  -- Document decisions that are trivially reversible
  -- Just list the decision without context or reasoning
  -- Are so long they are not read
  -- Are written after the fact to justify a decision already made
```

### When to write an ADR

```
Always write an ADR when:
  -- Choosing a technology that will be hard to replace (database, framework)
  -- Adopting a new architectural pattern for the service
  -- Making a significant trade-off with known downsides
  -- The team disagreed and a decision was made anyway (document the reasoning)

Consider writing an ADR when:
  -- Someone asks "why did we do it this way?"
  -- The decision affects more than one team
  -- The decision deviates from the Telia standard approach

Do NOT write an ADR for:
  -- Trivial implementation details
  -- Decisions that can be changed in an afternoon
  -- Things that are documented in the code itself
```

### ADR numbering

ADRs are numbered sequentially per service: ADR-001, ADR-002, etc.
The Architecture Doc Agent assigns the next number when CREATE_ADR is run.
Numbers are never reused -- deprecated ADRs keep their number.
