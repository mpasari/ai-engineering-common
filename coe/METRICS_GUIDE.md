## 1. Purpose

This guide defines how teams measure AI-assisted engineering adoption
and how the CoE reports on commons usage across Telia. Metrics give
the CoE signal on what is working, what is underused, and where to
focus improvement effort.

---

## 2. Team-level metrics

Champions collect these monthly using the Jira queries in CHAMPION_GUIDE.md
section 3.4.

### 2.1 Core adoption metrics

| Metric | Jira query basis | Target (mature team) |
|---|---|---|
| AI-generated stories (%) | labels = "ai-generated", issuetype = Story, status = Done | > 60% |
| AI-reviewed PRs (%) | labels = "ai-reviewed", issuetype = Story | > 80% |
| Gate compliance (%) | labels = "hitl-approved" / total AI-generated | 100% |
| Spec coverage (%) | Stories with linked spec vs total stories | > 70% |

### 2.2 Quality signal metrics

| Metric | Source | Signal |
|---|---|---|
| BLOCK findings per PR | Peer Review Agent output | Declining = improving code quality |
| AC pass rate | Feature Validation Agent | Should be > 90% on first run |
| Known error recurrence | KEDB occurrence counter | Declining = Problem Mgmt working |
| Onboarding time | Days to first merged PR | Declining = onboarding agent working |

### 2.3 How to report

Champions send a monthly one-pager to the CoE using this format:

```
Team: [Team name]
Month: [Month Year]
Engineers using commons: [N] of [N]

Core metrics:
  AI-generated stories:  [N]%
  AI-reviewed PRs:       [N]%
  Gate compliance:       [N]%

Top 3 commands used this month:
  1. [Command name]
  2. [Command name]
  3. [Command name]

What worked well:
  [One sentence]

What needs improvement:
  [One sentence]

Feedback for CoE:
  [Any issues, gaps, or suggestions]
```

## 3. CoE-level metrics

CoE Core aggregates team metrics quarterly and reports to management.

### 3.1 Programme metrics

| Metric | Calculation | Target |
|---|---|---|
| Teams using commons | Teams with npm install in last 30 days | All Telia engineering teams |
| Commons version currency | % teams on latest minor version | > 80% |
| Contribution rate | PRs merged per quarter (non-CoE-Core) | > 5 per quarter |
| Champion coverage | Teams with active champion / total teams | 100% |

### 3.2 Commons health metrics

| Metric | Source | Target |
|---|---|---|
| Agent accuracy (self-reported) | Champion feedback | > 85% satisfactory |
| HITL gate bypass incidents | Security audit | 0 |
| Foundation file review cadence | File review dates | All reviewed quarterly |
| Open issues (unresolved > 30 days) | GitHub issues | < 5 |

## 4. What not to measure

Some metrics look useful but create bad incentives:

```
Do NOT measure:
  -- Lines of code generated by AI
    (encourages over-generation, bloated PRs)

  -- Speed of story completion for AI-assisted stories
    (encourages skipping HITL gates)

  -- Number of commands used per engineer
    (encourages using commands for the sake of it)

  -- Percentage of code that is AI-generated
    (code ownership and understanding matter more than origin)
```
