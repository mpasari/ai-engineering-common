## 1. Purpose

This file defines how context is transferred between agents, between
agent sessions, and at HITL gates. It ensures that any agent instance
can resume a task from exactly the point where another agent or session
left off, without loss of context or duplication of work.

Every agent in the ai-engineering-commons system uses this protocol
when:
- Passing a task to a specialist agent (Orchestrator to Code Gen, etc.)
- Pausing at a HITL gate and saving state
- Resuming a task after HITL gate approval
- Handing off to the next agent in a journey flow
- Ending a session mid-task

Referenced by:
- `HITL_PROTOCOL.md` section 3.3 -- state saving at gates
- `agents/ORCHESTRATOR_AGENT.md` -- primary user of this protocol
- All agent skill files -- each defines its own handover fields
- `foundation/MEMORY_MANAGEMENT.md` -- how state is stored persistently

## 2. Handover principles

### 2.1 Assume nothing is remembered

Every handover package must be self-contained. The receiving agent
cannot rely on any shared memory, session history, or ambient context.
If a fact is needed to continue the task, it must be in the handover
package.

### 2.2 State over narrative

The handover package is structured data, not a summary paragraph.
Structured fields are parseable by agents and auditable by the
Compliance Agent. Narrative summaries are supplementary, not primary.

### 2.3 Minimum viable context

Include what is needed to continue, nothing more. Large handover
packages slow down context loading and increase the risk of the
receiving agent focusing on irrelevant detail. If a piece of
information is not needed to complete the remaining work, leave it out.

### 2.4 Explicit next action

The handover package always specifies the exact next action the
receiving agent should take. The receiving agent does not need to
infer what to do next from the context -- it is told explicitly.

## 3. Standard handover package format

Every handover, regardless of agent type or task complexity, uses this
format. Fields marked REQUIRED must always be present. Fields marked
OPTIONAL are included when applicable.

```
=== AGENT HANDOVER PACKAGE ===

-- IDENTITY --
Handover ID:      [UUID -- unique identifier for this handover]
Generated by:     [Agent name] (commons v[version])
Generated at:     [ISO 8601 UTC timestamp]
Receiving agent:  [Agent name, or "HUMAN" if returning to human]
Journey flow:     [Flow ID e.g. J03, or "none"]
Jira ticket:      [Primary ticket number, or "none"]

-- TASK CONTEXT --
Task summary:     [One sentence -- what this task is trying to achieve]
Task started:     [ISO 8601 UTC timestamp when the overall task began]
Requested by:     [Role of the person who initiated the task]
Priority:         [P0 / P1 / P2 / P3 -- from the Jira ticket]

-- COMPLETED WORK --
[Numbered list of everything that has been done. Be specific.
 Include file names, ticket numbers, Confluence page URLs,
 PR numbers, and the outcome of each action.]

1. [Action taken] -- [Outcome] -- [Reference/URL]
2. [Action taken] -- [Outcome] -- [Reference/URL]
...

-- CURRENT STATE --
[The exact state of the task at the moment of handover. What exists,
 what has changed, what is in progress and incomplete.]

-- DECISIONS MADE --
[List of decisions that have already been made and must not be
 revisited unless explicitly flagged. Include who made each decision.]

1. Decision: [What was decided]
   Made by:  [Role]
   Reason:   [One sentence]

-- OPEN QUESTIONS --
[Any unresolved questions or ambiguities the receiving agent should
 be aware of. If none, write "None".]

-- CONSTRAINTS ACTIVE --
[Any constraints from AGENT.md, HITL_PROTOCOL.md, or other files
 that are specifically relevant to the remaining work. Do not list
 all constraints -- only those the receiving agent needs to be
 actively aware of for this specific task.]

-- NEXT ACTION --
Next agent:   [Agent name or "HUMAN"]
Next action:  [Exact description of what the receiving agent or human
               must do first. One action only -- not a list.]
Gate if any:  [HITL gate ID if the next action involves a gate, or "none"]

-- FILES AND REFERENCES --
[List of all files, tickets, pages, and PRs that the receiving agent
 will need. Include full paths or URLs where possible.]

- [File type]: [Path or URL]
- [File type]: [Path or URL]
...

-- HANDOVER NOTES --
[Optional free text for anything that does not fit the structured
 fields above. Keep brief. If empty, omit this section.]

=== END HANDOVER PACKAGE ===
```

## 4. Handover examples

### 4.1 Orchestrator to Spec Writer Agent (Journey J03 -- New feature)

```
=== AGENT HANDOVER PACKAGE ===

-- IDENTITY --
Handover ID:      h-2025-01-15-001
Generated by:     Orchestrator Agent (commons v1.0.0)
Generated at:     2025-01-15T09:15:00Z
Receiving agent:  Spec Writer Agent
Journey flow:     J03
Jira ticket:      PROJ-412

-- TASK CONTEXT --
Task summary:     Generate technical spec for order cancellation feature
Task started:     2025-01-15T09:00:00Z
Requested by:     Tech Lead
Priority:         P2

-- COMPLETED WORK --
1. Read Jira story PROJ-412 -- Story has ACs, design link, no open dependencies -- PROJ-412
2. Checked DEPENDENCY_MAPPER -- No conflicting specs found in Confluence -- Clean
3. Confirmed story is in Ready state -- Tech Lead confirmed readiness -- Jira status

-- CURRENT STATE --
Story PROJ-412 is ready for spec generation. The story describes adding
a cancel endpoint to the Orders service. The AC requires that cancellation
is only permitted for orders in PENDING or PROCESSING status, and that
a cancellation reason is mandatory. No existing spec covers this endpoint.

-- DECISIONS MADE --
1. Decision: Spec will be written for Orders service v1 API, not v2
   Made by:  Tech Lead (confirmed in Jira comment 2025-01-15)
   Reason:   V2 migration is not scheduled until Q3

-- OPEN QUESTIONS --
1. Should cancelled orders be soft-deleted or status-updated only?
   (Ask Tech Lead at gate C01 if not resolvable from existing specs)

-- CONSTRAINTS ACTIVE --
- Security gate C05 applies if spec includes any auth changes
- Monetary amounts must use string representation per API_DESIGN_STANDARDS.md
- Maximum nesting depth 2 per API_DESIGN_STANDARDS.md section 2.1

-- NEXT ACTION --
Next agent:   Spec Writer Agent
Next action:  Generate technical spec for PROJ-412 using TECHNICAL_SPEC_TEMPLATE.md.
              Read existing Orders service spec in Confluence before generating
              to avoid conflicts with existing endpoints.

-- FILES AND REFERENCES --
- Jira story:          https://telia.atlassian.net/browse/PROJ-412
- Confluence space:    https://telia.atlassian.net/wiki/spaces/ORDERS

- Spec template:       sdlc/spec/TECHNICAL_SPEC_TEMPLATE.md
- HITL gate ref:       foundation/HITL_PROTOCOL.md (gate C01)

=== END HANDOVER PACKAGE ===
```

### 4.3 Human approval back to agent (resuming after gate)

When a human approves at a gate, they provide the approval signal defined
in the HITL gate output. The Orchestrator Agent then creates a resumption
handover to the next agent in the flow:

```
=== AGENT HANDOVER PACKAGE ===

-- IDENTITY --
Handover ID:      h-2025-01-15-003
Generated by:     Orchestrator Agent (commons v1.0.0)
Generated at:     2025-01-15T14:00:00Z
Receiving agent:  Code Gen Agent
Journey flow:     J03
Jira ticket:      PROJ-412

-- TASK CONTEXT --
Task summary:     Generate code for order cancellation feature (spec approved)
Task started:     2025-01-15T09:00:00Z
Requested by:     Tech Lead
Priority:         P2

-- COMPLETED WORK --
1. Story PROJ-412 read and understood
2. Technical spec generated and saved to Confluence (page 12345)
3. Gate C01 approved by Tech Lead (Erik Larsen) at 2025-01-15T13:45:00Z

-- CURRENT STATE --
Technical spec is approved. Ready for code generation. No outstanding
questions or concerns from the Tech Lead review.
Tech Lead confirmed: use status-update to CANCELLED, not soft-delete.

-- DECISIONS MADE --
1. Decision: Cancellation uses status update to CANCELLED
   Made by:  Tech Lead (Erik Larsen) -- confirmed at gate C01
   Reason:   Consistent with existing Orders service pattern

2. Decision: New endpoint POST /api/v1/orders/{orderId}/cancel
   Made by:  Spec Writer Agent, confirmed by Tech Lead at gate C01
   Reason:   Per API_DESIGN_STANDARDS.md action endpoint convention

-- OPEN QUESTIONS --
None.

-- CONSTRAINTS ACTIVE --
- Follow Backend Patterns from sdlc/engineering/BACKEND_PATTERNS.md
- Security review required before PR is raised (gate D02 if findings)
- HITL gate D01 required before merge -- Tech Lead approval on PR

-- NEXT ACTION --
Next agent:   Code Gen Agent
Next action:  Generate the cancellation endpoint implementation per the
              approved spec at Confluence page 12345. Start with the
              domain method Order.cancel() before the controller layer.
Gate if any:  none -- generate freely until PR is ready, then gate D01

-- FILES AND REFERENCES --
- Approved spec (Confluence): https://telia.atlassian.net/wiki/spaces/ORDERS/pages/12345
- Jira story:                 https://telia.atlassian.net/browse/PROJ-412
- Backend patterns:           sdlc/engineering/BACKEND_PATTERNS.md
- Security standards:         foundation/SECURITY_STANDARDS.md
- PR standards:               sdlc/engineering/PR_STANDARDS.md
- Gate C01 approval log:      Jira PROJ-412 comment (2025-01-15T13:45:00Z)

=== END HANDOVER PACKAGE ===
```

## 5. Handover for interrupted sessions

When a task is interrupted mid-execution (session ends, context window
limit reached, agent error), the agent must produce an interruption
handover before stopping. This differs from a gate handover in that
the work is incomplete rather than paused intentionally.

```
=== AGENT HANDOVER PACKAGE (INTERRUPTED) ===

-- IDENTITY --
Handover ID:      h-2025-01-15-004-INT
Generated by:     Code Gen Agent (commons v1.0.0)
Generated at:     2025-01-15T16:30:00Z
Receiving agent:  Code Gen Agent (new session)
Journey flow:     J03
Jira ticket:      PROJ-412

-- INTERRUPTION REASON --
Context window limit reached during code generation. Task is incomplete.

-- TASK CONTEXT --
Task summary:     Generate order cancellation feature code
Task started:     2025-01-15T14:00:00Z
Requested by:     Tech Lead
Priority:         P2

-- COMPLETED WORK --
1. Generated Order.cancel() domain method -- complete -- OrderEntity.java line 87
2. Generated OrderApplicationService.cancelOrder() -- complete -- OrderApplicationService.java
3. Generated CancelOrderRequest DTO -- complete -- api/dto/CancelOrderRequest.java
4. Started OrderController.cancelOrder() endpoint -- INCOMPLETE -- stopped at line 12

-- CURRENT STATE --
OrderController.cancelOrder() is partially written. The method signature
and @PostMapping annotation are in place but the body is not complete.
The file currently has a compilation error due to the incomplete method.
All other files generated in this session compile correctly.

-- DECISIONS MADE --
1. Decision: Cancellation reason is mandatory (not optional)
   Made by:  Code Gen Agent (from spec -- ACs state reason is required)
   Reason:   AC-3 in PROJ-412 explicitly states reason field is mandatory

-- OPEN QUESTIONS --
1. Should the cancellation reason have a maximum length constraint?
   Not specified in spec. Recommend 500 characters -- confirm with Tech Lead
   if time permits, otherwise apply 500 chars as a safe default.

-- CONSTRAINTS ACTIVE --
- File OrderController.java currently has a compilation error -- fix first
- Do not regenerate already-complete files -- they are correct
- Security review still required before PR

-- NEXT ACTION --
Next agent:   Code Gen Agent (new session)
Next action:  Complete OrderController.cancelOrder() method body starting
              from the existing method signature. Then generate the unit
              tests for all four new classes.
Gate if any:  none until PR is ready (then gate D01)

-- FILES AND REFERENCES --
- Approved spec:      https://telia.atlassian.net/wiki/spaces/ORDERS/pages/12345
- Jira story:         https://telia.atlassian.net/browse/PROJ-412
- Incomplete file:    src/main/java/com/telia/orders/api/OrderController.java
- Complete files:
    src/main/java/com/telia/orders/domain/OrderEntity.java (line 87)
    src/main/java/com/telia/orders/application/OrderApplicationService.java
    src/main/java/com/telia/orders/api/dto/CancelOrderRequest.java

=== END HANDOVER PACKAGE (INTERRUPTED) ===
```

## 6. Where handover packages are stored

| Scenario | Storage location |
|---|---|
| HITL gate handover | Jira ticket comment on the primary ticket |
| Agent-to-agent handover | In-memory context passed directly (not stored) |
| Interruption handover | Jira ticket comment + Git commit message if applicable |
| Multi-day task state | Confluence page in the project space, linked from Jira |
| SRE Agent Tier 3/4 escalation | Jira incident ticket + PagerDuty alert body |
| Post-incident handover | Post-mortem Confluence page |

Handover packages stored in Jira comments must be in a collapsible
section so they do not clutter the ticket view for human readers.

## 7. Receiving a handover package

When an agent receives a handover package, it must:

1. Confirm receipt:
   ```
   HANDOVER RECEIVED
   Handover ID: [ID from package]
   From:        [Generating agent]
   Task:        [Task summary]
   Starting:    [Next action from package]
   ```

2. Verify completeness -- all REQUIRED fields are present.

3. Verify consistency -- the completed work list matches reality
   (files exist where stated, tickets are in the stated status).
   If any inconsistency is found, flag it before proceeding.

4. Load all referenced files listed in the FILES AND REFERENCES section.

5. Execute the NEXT ACTION exactly as specified.

6. If the NEXT ACTION leads to a HITL gate, apply HITL_PROTOCOL.md
   section 3 immediately.
