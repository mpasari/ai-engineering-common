## 1. Purpose

This file defines how every agent in the ai-engineering-commons system
interacts with GitHub. It covers repository conventions, branch rules,
pull request standards, GitHub Actions integration, Copilot configuration
generation, and the security boundaries that govern all agent access to
source code repositories.

- `agents/RELEASE_AGENT.md` -- release and changelog management

## 4. Branch strategy

### 4.1 Branch naming

All branches follow this naming convention. Agents must create branches
with exactly this format -- no variations.

```
feature/{jira-ticket-key}-{short-description}
fix/{jira-ticket-key}-{short-description}
refactor/{jira-ticket-key}-{short-description}
chore/{jira-ticket-key}-{short-description}
release/{version}
hotfix/{jira-ticket-key}-{short-description}

Examples:
  feature/PROJ-412-order-cancellation
  fix/PROJ-398-token-refresh-race-condition
  refactor/PROJ-445-extract-payment-domain-service
  chore/PROJ-501-upgrade-spring-boot-3-3
  release/1.5.0
  hotfix/PROJ-521-auth-null-pointer
```

Rules:

- All lowercase
- Hyphens only -- no underscores, no slashes within the description
- Maximum 60 characters total
- Jira ticket key always included
- Short description is kebab-case, max 40 characters

### 4.2 Protected branches


| Branch pattern | Protection level    | Who can merge                             |
| -------------- | ------------------- | ----------------------------------------- |
| `main`         | Full protection     | CoE core (commons) / Tech Lead (projects) |
| `release/*`    | Full protection     | Tech Lead + DevOps                        |
| `develop`      | Require PR + review | Any team member                           |
| `feature/*`    | No protection       | Agent + engineer                          |
| `fix/*`        | No protection       | Agent + engineer                          |


Full protection means:

- Require PR with minimum 2 approvals (commons) or 1 approval (projects)
- Require all status checks to pass
- Require branch to be up to date before merge
- No force pushes
- No deletions

### 4.3 Agent branch operations

Agents always work on feature or fix branches. They never commit directly
to protected branches. The sequence is:

```
1. Agent creates branch from main (or develop if used)
   Branch name follows section 4.1 conventions

2. Agent commits work to the feature branch
   Commit messages follow CODING_STANDARDS.md section 6.1

3. Agent opens PR from feature branch to main
   PR follows standards in section 5

4. Human reviews and approves (HITL gate D01)

5. Human merges the PR (squash merge by default)
   Agent does not merge PRs
```

## 5. Pull request standards

### 5.1 PR title format

PR titles follow the same Conventional Commits format as commit messages:

```
{type}({scope}): {short description}

Examples:
  feat(orders): add order cancellation endpoint
  fix(auth): prevent token refresh race condition
  refactor(payment): extract domain service layer
  chore(deps): upgrade spring-boot to 3.3.0
  docs(orders): add cancellation API documentation
```

### 5.2 PR description template

The Peer Review Agent generates this description automatically from
the diff and commit messages. Engineers may edit it before requesting
review.

```markdown
## Summary
[What this PR does in 2-3 sentences]

## Motivation
[Why this change is needed -- link to Jira story or bug report]

## Changes
- [File or component]: [What changed and why]
- [File or component]: [What changed and why]

## Type of change
- [ ] Bug fix (non-breaking change that fixes an issue)
- [ ] New feature (non-breaking change that adds functionality)
- [ ] Breaking change (fix or feature that causes existing behaviour to change)
- [ ] Refactor (no functional change)
- [ ] Documentation update
- [ ] Dependency update

## How to test
[Steps to verify the change works correctly]

## Security considerations
[Any security implications -- or "None identified by Security Review Agent"]

## Screenshots (if UI change)
[Before and after screenshots -- or "Not applicable"]

## Checklist
- [ ] Code follows CODING_STANDARDS.md
- [ ] Tests added or updated
- [ ] Security review passed (Security Review Agent)
- [ ] Accessibility check passed (if UI change)
- [ ] Documentation updated in Confluence
- [ ] CHANGELOG.md updated (if applicable)

---
PR generated by: [Agent name] (commons v[version])
Jira ticket: [URL]
Confluence spec: [URL or "Not applicable"]
```

### 5.3 PR labels

Agents apply labels to PRs to enable filtering and routing:


| Label                    | When applied                                                 |
| ------------------------ | ------------------------------------------------------------ |
| `ai-generated`           | PR created by an agent                                       |
| `ai-reviewed`            | Peer Review Agent has completed review                       |
| `security-reviewed`      | Security Review Agent has completed review                   |
| `accessibility-reviewed` | Accessibility Agent has completed review                     |
| `awaiting-human-review`  | Agent review complete, human approval needed                 |
| `breaking-change`        | PR contains a breaking change                                |
| `dependency-update`      | PR updates a dependency                                      |
| `hotfix`                 | PR is a hotfix to production                                 |
| `do-not-merge`           | PR must not be merged without explicit removal of this label |


### 5.4 PR size limits

See `CODING_STANDARDS.md` section 6.3 for PR size guidance. The Peer
Review Agent flags PRs that exceed 1000 lines as requiring Tech Lead
justification.

### 5.5 Required status checks

All PRs to `main` must pass these status checks before merge:

```yaml
# Required checks -- all must pass
required_status_checks:
  strict: true
  contexts:
    - build
    - unit-tests
    - integration-tests
    - security-scan
    - lint
    - dependency-check
```

## 6. Commit conventions

All commits follow Conventional Commits format as defined in
`CODING_STANDARDS.md` section 6.1. Agents always produce compliant
commit messages. The Release Agent uses commit history to generate
changelogs automatically.

### 6.1 Agent commit signature

Every commit made by an agent includes a trailer identifying the agent:

```
feat(orders): add order cancellation endpoint

Implements POST /api/v1/orders/{orderId}/cancel with mandatory
reason field. Publishes OrderCancelledEvent to Kafka.

Closes PROJ-412

Agent: Code Gen Agent (commons v1.0.0)
Confidence: High
```

The `Agent:` and `Confidence:` trailers are machine-readable and used
by the audit trail system defined in `COMPLIANCE_STANDARDS.md`.

Confidence levels:

- `High` -- agent is confident the output is correct
- `Medium` -- agent has some uncertainty, recommends careful review
- `Low` -- agent is not confident, human review is essential

### 6.2 Commit signing

All agent commits must be signed using the agent service account GPG key.
This allows the audit trail to distinguish agent commits from human commits.

## 7. GitHub Actions integration

### 7.1 Required workflows

The Pipeline Agent generates and maintains these workflows for every
service repository. They must be present and passing before any PR
can merge.

**ci.yml -- continuous integration:**

```yaml
name: CI

on:
  push:
    branches: ['feature/**', 'fix/**', 'refactor/**', 'chore/**']
  pull_request:
    branches: [main, develop]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Java
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
      - name: Build
        run: ./mvnw clean compile
      - name: Unit tests
        run: ./mvnw test
      - name: Integration tests
        run: ./mvnw verify -P integration-tests
      - name: Security scan
        uses: ./.github/actions/security-scan
      - name: Lint
        run: ./mvnw checkstyle:check
      - name: Dependency check
        run: ./mvnw dependency-check:check
```

**security-scan action (.github/actions/security-scan/action.yml):**

```yaml
name: Security scan
runs:
  using: composite
  steps:
    - name: SAST scan
      run: |
        # SpotBugs with FindSecBugs plugin
        ./mvnw spotbugs:check
    - name: Secrets scan
      uses: trufflesecurity/trufflehog@main
      with:
        path: ./
        base: main
    - name: Dependency vulnerability scan
      run: ./mvnw org.owasp:dependency-check-maven:check
      shell: bash
```

### 7.2 Permitted workflow triggers agents may use

The Pipeline Agent may trigger these workflow dispatch events via
T-GIT-07:


| Workflow                | Trigger condition             | HITL gate required            |
| ----------------------- | ----------------------------- | ----------------------------- |
| `ci.yml`                | Code commit to feature branch | No                            |
| `security-scan.yml`     | Manual or scheduled           | No                            |
| `integration-test.yml`  | Pre-PR validation             | No                            |
| `staging-deploy.yml`    | PR merged to develop          | Gate A02 (Tech Lead)          |
| `production-deploy.yml` | Release tag created           | Gate A02 (Tech Lead + DevOps) |


Agents may NOT trigger production deployment workflows autonomously.
A production deployment always requires a human to create the release
tag, which then triggers the deployment workflow.

### 7.3 GitHub Actions secrets

Agents do not have direct access to GitHub Actions secrets. Secrets
are managed by DevOps and injected into workflows at runtime. Agents
that need credentials read them from Azure Key Vault via the service
account, not from GitHub secrets directly.


| Secret name in GitHub  | Purpose                 | Managed by              |
| ---------------------- | ----------------------- | ----------------------- |
| `AZURE_CLIENT_ID`      | Azure Managed Identity  | DevOps                  |
| `JIRA_API_TOKEN`       | Jira API access         | DevOps (from Key Vault) |
| `CONFLUENCE_API_TOKEN` | Confluence API access   | DevOps (from Key Vault) |
| `PACKAGE_TOKEN`        | GitHub Packages publish | DevOps                  |
| `SONAR_TOKEN`          | SonarCloud analysis     | DevOps                  |

## 8. Copilot instructions file generation

The `aec update` CLI command generates `.github/copilot-instructions.md`
by merging content from the commons and the project layer. This file
is what GitHub Copilot reads to understand the project context.

### 8.1 Merge order

```
1. foundation/AGENT.md                    -- agent identity and constraints
2. foundation/CODING_STANDARDS.md         -- coding conventions
3. foundation/SECURITY_STANDARDS.md       -- security requirements (summary)
4. foundation/API_DESIGN_STANDARDS.md     -- API conventions (summary)
5. .ai/project/ARCHITECTURE_OVERVIEW.md  -- this project's architecture
6. .ai/project/MODULE_REGISTRY.md        -- this project's modules
7. .ai/project/OVERRIDES/CODING_STANDARDS.md -- project overrides if present
```

### 8.2 Copilot instructions file structure

The generated file follows this structure so Copilot can parse it
effectively:

```markdown
# [Project name] -- AI Engineering Context

## Agent identity and constraints
[Content from AGENT.md sections 2, 3, 4]

## Coding standards
[Content from CODING_STANDARDS.md -- condensed to key rules]

## Security requirements
[Content from SECURITY_STANDARDS.md section 8 checklist only]

## API design conventions
[Content from API_DESIGN_STANDARDS.md sections 2 and 4 only]

## This project

### Architecture overview
[Content from .ai/project/ARCHITECTURE_OVERVIEW.md]

### Module registry
[Content from .ai/project/MODULE_REGISTRY.md]

### Project-specific overrides
[Content from .ai/project/OVERRIDES/ if present]
```

### 8.3 CLAUDE.md generation

`CLAUDE.md` is generated for Claude Code and contains more context
than the Copilot instructions file because Claude Code supports a
larger context window:

```
Merged in order:
1. foundation/AGENT.md                    -- full content
2. foundation/CODING_STANDARDS.md         -- full content
3. foundation/SECURITY_STANDARDS.md       -- full content
4. foundation/API_DESIGN_STANDARDS.md     -- full content
5. .ai/project/ARCHITECTURE_OVERVIEW.md  -- full content
6. .ai/project/MODULE_REGISTRY.md        -- full content
7. .ai/project/INTEGRATION_MAP.md        -- full content
8. .ai/project/DATA_MODEL.md             -- full content (if present)
9. .ai/project/OVERRIDES/               -- any override files
```

## 11. Rate limiting and error handling


| Response code           | Meaning                                      | Agent action                                                                                                             |
| ----------------------- | -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| 403 Forbidden           | Rate limit hit or insufficient permissions   | Check X-RateLimit-Remaining header. If rate limit: wait until X-RateLimit-Reset. If permissions: stop and flag to human. |
| 404 Not Found           | Repo, branch, or file does not exist         | Stop, flag to human -- do not create silently                                                                            |
| 409 Conflict            | Merge conflict or concurrent write           | Fetch latest, resolve conflict, retry                                                                                    |
| 422 Unprocessable       | Invalid request (e.g. branch already exists) | Log, adjust request, retry once                                                                                          |
| 503 Service Unavailable | GitHub temporarily unavailable               | Retry with exponential backoff (max 3)                                                                                   |


### 11.1 Rate limit management

GitHub API rate limits apply per token:

- Authenticated requests: 5000 per hour
- Search API: 30 requests per minute
- GraphQL API: 5000 points per hour

Agents check `X-RateLimit-Remaining` on every response and pause
when remaining is below 100 requests, waiting until `X-RateLimit-Reset`.

## 12. Agent service account


| Setting         | Value                                                                                                    |
| --------------- | -------------------------------------------------------------------------------------------------------- |
| GitHub username | ai-engineering-agent                                                                                     |
| Account type    | GitHub App (preferred) or machine user                                                                   |
| Token storage   | Azure Key Vault -- secret name: github-agent-token                                                       |
| Token type      | Fine-grained PAT or GitHub App installation token                                                        |
| Token rotation  | Monthly (PAT) or automatic (GitHub App)                                                                  |
| Permissions     | Contents: read/write, Pull requests: read/write, Actions: read/write, Issues: read/write, Metadata: read |


**GitHub App is strongly preferred over PAT** because:

- Tokens are short-lived (1 hour) and automatically rotated
- Permissions are scoped per repository, not per organisation
- Audit logs show the App name, not a generic service account name
- Revocation is immediate and does not affect human accounts
