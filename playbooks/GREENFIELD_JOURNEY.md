## Overview

This playbook walks you through creating a new service from scratch
using the commons. It covers everything from the initial scaffold to
the first story being merged. Estimated time for a prepared team: 2-3 days.

**Tools needed:** Claude Code (with Jira + Confluence MCP), VS Code + Copilot
**Prerequisites:** SECRETS_GUIDE.md and MCP_SETUP.md completed

---

## Phase 1 -- Architect approval (Day 1, morning)

Before any code is written, the Architect approves the new service.
This is gate B01 -- it cannot be skipped.

### Step 1.1 -- Prepare the service brief

Write a one-page brief covering:

```
Service name: [name] (e.g. order-cancellation-service)
Domain: [domain] (e.g. orders)
Purpose: [2-3 sentences -- what problem this solves]
Tech stack: [Java/TypeScript/C#]
Consumers: [who will call this service]
Dependencies: [what this service will call]
Team: [team name and tech lead]
```

### Step 1.2 -- Present gate B01 to Architect

In Claude Code:

```
Gate B01 for a new service:

Service brief:
[paste your brief here]

Please prepare the gate B01 presentation for Architect review.
```

Claude Code generates the gate B01 output which you share with
the Architect. The Architect approves via Jira comment or email.

### Step 1.3 -- Run SCAFFOLD_SERVICE

Once B01 is approved, in Claude Code:

```
SCAFFOLD_SERVICE name="[service-name]" domain="[domain]" stack="[java|typescript|csharp]" team="[team-name]"
```

**What happens automatically:**
- GitHub repository created at `telia-company/[service-name]`
- Standard folder structure seeded
- Commons installed and `npx aec init` run
- `.ai/project/` stubs created
- Initial ADRs drafted in Confluence (status: Proposed)
- CI/CD pipeline created
- First 5 foundation stories created in Jira
- Gate B05 presented for Architect to approve the ADRs

**What you do next:**
- Clone the new repository
- Fill in the `.ai/project/` stub files (see Phase 2)
- Approve gate B05 with the Architect

## Phase 2 -- Fill in project context (Day 1, afternoon)

The `.ai/project/` files need real content before any agent can
give useful project-specific output.

### Step 2.1 -- Fill in ARCHITECTURE_OVERVIEW.md

Open `.ai/project/ARCHITECTURE_OVERVIEW.md` and fill in:

```
## What this system does
[Your 2-3 sentence description]

## Technology stack
[Fill in the table with your actual stack and versions]

## System boundaries
[What does this service own? What does it not do?]

## Key constraints
[Any specific rules agents must follow when generating code for this service]
```

### Step 2.2 -- Fill in MODULE_REGISTRY.md

For a new service, you likely have these initial modules:

```
| Module name | Path | Language | Owner | Status | Description |
|---|---|---|---|---|---|
| [service]-api | src/main/java/.../api | Java | [team] | Active | REST controllers and DTOs |
| [service]-application | src/main/java/.../application | Java | [team] | Active | Use cases and orchestration |
| [service]-domain | src/main/java/.../domain | Java | [team] | Active | Entities, business rules |
| [service]-infrastructure | src/main/java/.../infrastructure | Java | [team] | Active | Repositories, clients |
```

### Step 2.3 -- Fill in INTEGRATION_MAP.md

List any integrations you know about upfront. You can add more later.

### Step 2.4 -- Regenerate tool configs

```powershell
npx aec update
git add .github/copilot-instructions.md CLAUDE.md .cursorrules
git commit -m "chore: fill in project-layer files and regenerate tool configs"
git push
```

### Step 2.5 -- Verify context quality

In Claude Code:

```
What project is this? Describe the architecture and list the modules.
```

Expected: Claude describes your service accurately from the project-layer files.
If it gives generic output, the files are not filled in enough -- go back to Step 2.1.

## Phase 3 -- Architect reviews ADRs (Day 1, end of day)

Gate B05: Architect reviews the three initial ADRs generated by
SCAFFOLD_SERVICE (technology stack, database, API approach).

In Claude Code (or share the Confluence URLs directly):

```
Show the gate B05 output for [service-name] with the Confluence ADR URLs.
```

Share the ADR pages with the Architect. They update each ADR from
"Proposed" to "Accepted" status in Confluence.

## Phase 4 -- First sprint (Day 2)

With the service scaffolded and context loaded, pick up the first
Jira story (Story 1: local development environment setup).

### Step 4.1 -- Explain the domain before coding

Before any engineering work, orient yourself:

```
# In Claude Code:
EXPLAIN_MODULE [service]-domain
```

For a new service this is brief, but the habit of running EXPLAIN_MODULE
before touching any module is important -- it becomes critical for
brownfield work.

### Step 4.2 -- Write the spec for Story 1

Story 1 is always "Set up local development environment". It has clear ACs
already created by SCAFFOLD_SERVICE.

```
# In Claude Code:
WRITE_SPEC [STORY-1-KEY]
```

Claude Code reads the story from Jira, generates a technical spec in
Confluence, and presents gate C01 for Tech Lead review.

**Gate C01:** Tech Lead reviews the spec and approves via Jira comment.
For the local dev setup story, the spec is simple -- approval takes minutes.

### Step 4.3 -- Generate code

```
# In Claude Code:
GENERATE_CODE [STORY-1-KEY]
```

Claude Code reads the approved spec, generates the docker-compose.yml,
health check endpoint, and supporting configuration. It commits to
a feature branch and opens a PR.

### Step 4.4 -- Review the PR

```
# In Claude Code:
REVIEW_PR [PR-NUMBER]
```

Claude Code runs the full peer review: coding standards, security
checklist, secrets scan. For a new service foundation story, this
is usually clean.

**Gate D01:** Tech Lead reviews the PR in GitHub and approves.

### Step 4.5 -- Validate the story

```
# In Claude Code:
VALIDATE_STORY [STORY-1-KEY]
```

Claude Code verifies all ACs: starts the docker-compose environment,
calls the health check endpoint, asserts on the response. All ACs
should pass.

Story transitions to Done.

## Phase 5 -- Observability setup (Day 2, afternoon)

Before deploying to any environment, set up monitoring.

```
# In Claude Code:
SETUP_OBSERVABILITY [service-name]
```

**What happens:**
- Grafana overview and SLO dashboards created
- Alert rules created for error rate, latency, availability
- Confluence runbooks created per alert
- SRE Dashboard Registry PR opened (gate D01 for Tech Lead)

Once the Registry PR is merged, the SRE Agent begins monitoring
this service automatically.

---

## Phase 6 -- Repeat for remaining stories (Day 3+)

The foundation stories follow the same WRITE_SPEC -> GENERATE_CODE
-> REVIEW_PR -> VALIDATE_STORY pattern. By Story 5 (authentication),
the team is comfortable with the flow.

### Typical first-sprint velocity

```
Story 1 (local dev): 1 day (includes setup overhead)
Story 2 (health check): 2 hours
Story 3 (CI pipeline): 3 hours
Story 4 (observability): 4 hours (includes SETUP_OBSERVABILITY)
Story 5 (authentication): 1 day (more complex -- auth integration test needed)
```

Total: approximately 3 days for the foundation sprint.

---

## Phase 7 -- First feature story

With the foundation in place, pick up the first real feature story.

The full flow for a feature story:

```
1. DRAFT_STORIES [feature brief]        -- if stories not yet created
2. ESTIMATE_STORIES [story key]         -- get story point suggestion
3. EXPLAIN_MODULE [affected module]     -- orient before coding
4. WRITE_SPEC [story key]              -- generate spec -> gate C01
5. GENERATE_MIGRATION [story key]      -- if DB changes needed -> gate C04
6. GENERATE_CODE [story key]           -- generate code -> PR
7. GENERATE_TESTS [story key]          -- ensure test coverage
8. REVIEW_PR [PR number]              -- automated review -> gate D01
9. VALIDATE_STORY [story key]         -- AC execution -> Done
10. REFRESH_ARCH_DOCS                 -- update architecture docs
```

This is journey flow J03 (new feature). For subsequent features the
setup overhead is gone -- the full flow takes hours not days.
