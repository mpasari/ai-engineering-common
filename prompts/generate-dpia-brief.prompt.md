Do not greet the user. Execute immediately.

You are the Data Processing Brief Agent. Your job is to produce a clear,
non-technical document that answers the questions a legal team, security
officer, enterprise customer, or DPA reviewer would ask about this system.

The audience is NOT technical. Avoid class names, API endpoints, and
implementation detail. Write in plain business language.

## Step 1 -- Read all project context

Read these files:
- .ai/project/JIRA_CONFIG.md            -- system name and team
- .ai/project/ARCHITECTURE_OVERVIEW.md  -- what the system does and its boundaries
- .ai/project/INTEGRATION_MAP.md        -- what external systems data flows to/from
- .ai/project/DATA_MODEL.md             -- what data is stored, what is PII
- .ai/project/MODULE_REGISTRY.md        -- system components
- .ai/project/TECH_DEBT_REGISTRY.md     -- known security or data risks

Also search Confluence for any existing privacy, security, or GDPR pages
in the configured space.

## Step 2 -- Generate the brief

Produce the document in this exact structure.
Use plain language throughout. No class names, no API paths, no technical jargon.
Write as if explaining to a senior lawyer or enterprise procurement officer.

---

# Data Processing and Security Brief
# [System name from ARCHITECTURE_OVERVIEW.md]
# Version: 1.0 | Generated: [date] | Owner: [team from JIRA_CONFIG.md]
# Status: DRAFT -- for review by Data Protection Officer and Tech Lead

---

## 1. What this system does

[2-3 sentences in plain language. What business problem does it solve?
Who uses it? What does it do with information provided to it?]

## 2. What data this system processes

### 2a. Data provided by end users

[What information does a user give to the system when they use it?
e.g. "Users provide natural language questions or requests. These may
contain names, account references, or other personal context depending
on what the user chooses to include."]

### 2b. Data retrieved or generated by the system

[What data does the system look up, generate, or return?
e.g. "The system retrieves account information from internal CRM systems
to answer customer questions. It generates text responses using an AI
language model."]

### 2c. Personal data categories

[List the categories of personal data the system may process.
Use GDPR Article 4 categories in plain language:
e.g. "Contact details, account identifiers, communication content"]

Do NOT list any special category data (health, biometric, political)
unless DATA_MODEL.md explicitly identifies it.

## 3. How data flows through the system

[Describe the journey of a piece of information in plain language.
A simple numbered flow:
1. User submits a request via [channel]
2. The request is sent to [what] for processing
3. [What] looks up relevant information from [where]
4. A response is generated and returned to the user
5. The interaction is logged for [what purpose] and retained for [how long]]

## 4. Where data is stored and for how long

| What is stored | Where | Retention period | Deleted how |
|---|---|---|---|
| [type of data] | [system/location] | [period] | [automatic/manual/on request] |

If retention periods are not documented in DATA_MODEL.md -- flag them:
"Retention period not yet defined -- requires Data Owner decision."

## 5. Data transfers and third-party processors

[For each external system in INTEGRATION_MAP.md, describe in plain language:
- What data is shared with it
- What it does with that data
- Where it is located (EU/non-EU)
- Whether a DPA is in place]

| Recipient | What data is shared | Purpose | Location | DPA status |
|---|---|---|---|---|
| [system name] | [data types] | [why] | [EU/Non-EU/Unknown] | [Yes/No/Unknown] |

For any row where DPA status is Unknown -- add:
"⚠ DPA status must be confirmed before processing personal data
through this integration."

## 6. Who has access to the data

[Describe access in business terms:
- Which teams can access what
- Whether access is logged
- Whether access is role-based
- Whether customers can request access logs]

## 7. Security controls

Describe the following controls in plain language.
Do not reference specific technologies unless necessary for clarity.

### 7a. Data in transit
[Is data encrypted when moving between systems? What standard?]

### 7b. Data at rest
[Is stored data encrypted? What standard?]

### 7c. Authentication and authorisation
[How is access controlled? Who can access what? Is MFA required?]

### 7d. Logging and audit trail
[What is logged? How long are logs kept? Who can access them?]

### 7e. Incident response
[Is there a documented process for data breaches? Who is notified?
Within what timeframe?]

## 8. AI and automated decision-making

[If the system uses AI or machine learning, answer these questions:
- Does the system make automated decisions that affect individuals?
- Can users request human review of an AI decision?
- Is the AI model trained on personal data?
- Where is the AI model hosted and by whom?
- What data is sent to the AI model and is it retained by the model provider?]

If the system does not use AI -- state: "This system does not use AI or
automated decision-making."

## 9. Customer confidentiality options

[What options are available to customers who have strict confidentiality
requirements? e.g.:
- Can data be isolated per customer?
- Can the system be deployed in a customer's own environment?
- Can AI processing be limited to EU-hosted models only?
- Can audit logs be provided to the customer on request?
- Can data be deleted on request within a specified timeframe?]

## 10. Known risks and open items

[List any data protection risks or open decisions from TECH_DEBT_REGISTRY.md
that are relevant to data handling. Write in plain language.
Flag anything that must be resolved before a DPA can be signed.]

| Risk or open item | Severity | Status | Owner |
|---|---|---|---|
| [description] | [High/Medium/Low] | [Open/In progress] | [team] |

## 11. Regulatory basis

[Which regulations apply to this system? e.g.:
- GDPR (EU 2016/679) -- if processing personal data of EU residents
- Norwegian Personal Data Act (personopplysningsloven)
- ePrivacy Directive -- if processing communication metadata
- NIS2 -- if the system is critical infrastructure
- DORA -- if the system supports financial services]

State the lawful basis for processing under GDPR Article 6 if known.
If not documented -- flag: "Lawful basis not yet formally documented --
requires Data Protection Officer review."

## 12. Contact and review

| Field | Value |
|---|---|
| Document owner | [team from JIRA_CONFIG.md] |
| Data Protection Officer | [DPO contact if known, otherwise: "Contact Security Lead"] |
| Tech Lead | [from JIRA_CONFIG.md if present] |
| Review cadence | Annually or when system changes significantly |
| Next review due | [date + 12 months] |
| Approved by | DRAFT -- pending DPO review |

## Step 3 -- Save to repository

Save the brief as a versioned governance artifact in the repository.
Do NOT publish to Confluence. Do NOT create any Jira issues.

File path: docs/governance/DPIA_BRIEF.md

If docs/governance/ does not exist -- create it.

The file must include a header block before the brief content:

```
# Data Processing and Security Brief
# System: [system name]
# Generated: [date]
# Commons version: [from .ai/project/COMMONS_VERSION.md]
# Status: DRAFT
# Review required by: Data Protection Officer, Security Lead, Tech Lead
#
# This file is a versioned governance artifact.
# Changes must go through PR review.
# Do not edit manually -- regenerate with /generate-dpia-brief when
# the system changes significantly.
```

Write the full brief content after the header block in clean Markdown.
Use standard Markdown formatting:
- ## and ### for headings
- - for bullet lists
- | for tables
- **bold** for labels
- ``` for code blocks

## Step 4 -- Tell the engineer

State:
1. The file path: docs/governance/DPIA_BRIEF.md
2. Which sections have gaps (missing retention periods, unknown DPA status etc.)
   These are the items the engineer must manually complete before sharing
3. Who needs to review it (DPO, Security Lead, Tech Lead)
4. Exact git commands to commit and share:

```
git add docs/governance/DPIA_BRIEF.md
git commit -m "docs: add DPIA brief for [system name]

Generated by /generate-dpia-brief from .ai/project/ context.
Status: DRAFT -- pending DPO and Security Lead review.
Gaps flagged: [list any missing retention periods or DPA status]"
git push origin [current branch]
```

5. Suggested next step: share the GitHub file URL with the legal or
   security team requesting the NDA/DPA assessment. They can comment
   directly on the file in GitHub or raise a PR with corrections.
