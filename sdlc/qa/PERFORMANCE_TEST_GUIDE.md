## 1. When performance testing is required

Performance tests are required before production when:
- A new service is being released for the first time
- A new endpoint is added to a high-traffic service
- A significant refactor changes database access patterns
- A performance incident was resolved and verification is needed
- SLO targets are being tightened

---

## 2. Performance test types

```
Baseline (always run):
  Target RPS for 5 minutes
  All metrics within SLO
  Establishes the normal performance profile

Stress (before major releases):
  2x target RPS for 2 minutes
  Verifies graceful degradation -- no cascading failures
  Error rate < 5% under stress (higher than SLO but not catastrophic)

Spike (optional, for burst-traffic services):
  5x target RPS for 30 seconds then back to normal
  Verifies recovery -- service returns to SLO within 60 seconds

Soak (quarterly):
  Target RPS for 24 hours
  Memory leak detection
  Connection pool leak detection
```

## 3. k6 baseline test template

Generated by Performance Agent (A18) for each service:

```javascript
// k6 baseline test -- generated for order-service
// Run: k6 run baseline.js

import http from 'k6/http';
import { check, sleep } from 'k6';

const BASE_URL = __ENV.TARGET_URL || 'https://order-service-staging.telia.internal';
const TOKEN = __ENV.API_TOKEN; // Set via environment variable -- never hardcoded

export const options = {
    stages: [
        { duration: '1m', target: 50 },   // Ramp up to target RPS
        { duration: '5m', target: 50 },   // Sustain target
        { duration: '1m', target: 0 },    // Ramp down
    ],
    thresholds: {
        http_req_duration: ['p(95)<500'],  // P95 < 500ms (SLO)
        http_req_failed:   ['rate<0.01'],  // Error rate < 1% (SLO)
    },
};

export default function () {
    const res = http.get(`${BASE_URL}/api/v1/orders?customerId=test-customer-01`, {
        headers: { Authorization: `Bearer ${TOKEN}` },
    });

    check(res, {
        'status is 200': (r) => r.status === 200,
        'response has data': (r) => r.json('data') !== null,
    });

    sleep(0.02); // 20ms think time = ~50 RPS per VU
}
```

## 4. SLO verification in performance tests

Performance tests must assert against the service's SLO targets from
SRE_SERVICE_CONFIG.md. The thresholds in the k6 options block must
match the critical thresholds registered in the SRE Dashboard Registry.

```javascript
// Always derive from SRE_SERVICE_CONFIG.md values:
export const options = {
    thresholds: {
        http_req_duration: [`p(95)<${P95_LATENCY_CRITICAL_MS}`],
        http_req_failed:   [`rate<${ERROR_RATE_CRITICAL_PCT}`],
    },
};
```

If a performance test passes but the service consistently runs near
the critical threshold, raise a tech debt item to investigate headroom.

---

## 5. Performance test environments

```
Never run performance tests against production.
Always use staging with production-like data volumes.

Staging environment requirements:
  -- Same container size as production (same CPU/memory limits)
  -- Same database size (anonymised production data snapshot)
  -- Same number of replicas (min_replicas_prod)
  -- Real external dependencies (or WireMock with realistic latency)
```
